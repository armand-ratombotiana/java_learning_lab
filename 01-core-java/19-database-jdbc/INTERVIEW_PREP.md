# Module 19: Database Access with JDBC - Interview Preparation

## 📝 Conceptual Questions

### Q1: What is the difference between `Statement` and `PreparedStatement`?
**Answer**:
- **Statement**: Used for executing static SQL queries. The SQL string is sent to the database and compiled every time it is executed. It is highly vulnerable to SQL Injection if user input is concatenated into the string.
- **PreparedStatement**: Extends `Statement`. It is used for executing parameterized queries (using `?` placeholders). The SQL query is pre-compiled by the database just once. The parameters are supplied later. This provides better performance for repeated queries and eliminates SQL Injection vulnerabilities because parameters are treated purely as data, not as executable code.

### Q2: What is a Connection Pool, and why is it used?
**Answer**:
Creating a physical connection to a database is an expensive and time-consuming operation (involving network handshakes, authentication, etc.). A **Connection Pool** (like HikariCP or Tomcat JDBC) maintains a cache of open, ready-to-use database connections. When the application needs to interact with the database, it borrows a connection from the pool. When finished, instead of closing the connection physically, it returns it to the pool for the next thread to use. This drastically improves application performance and scalability.

### Q3: Explain the concepts of `ResultSet` and `Cursor`.
**Answer**:
A `ResultSet` represents the table of data generated by executing a `SELECT` query. It maintains a **Cursor** pointing to its current row of data. Initially, the cursor is positioned *before* the first row. The `rs.next()` method moves the cursor to the next row and returns `true` if a row exists, allowing iteration through the results. 
Note: By default, `ResultSet` objects are forward-only and not updatable, though JDBC does support scrollable/updatable result sets if specified during Statement creation.

## 💻 Whiteboarding Scenarios

### Scenario 1: Refactoring for Security
**Problem**: The following login method is failing security audits. Why? Refactor it to be secure.

```java
public boolean login(String username, String password) throws SQLException {
    Connection conn = getConnection();
    Statement stmt = conn.createStatement();
    String query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'";
    ResultSet rs = stmt.executeQuery(query);
    return rs.next();
}
```

**Solution**:
It is vulnerable to SQL Injection. If a user enters `' OR '1'='1` as the username, the query evaluates to `WHERE username = '' OR '1'='1'`, bypassing the password check entirely.
Refactor using `PreparedStatement` and `try-with-resources`.

```java
public boolean login(String username, String password) throws SQLException {
    String query = "SELECT id FROM users WHERE username = ? AND password = ?";
    try (Connection conn = getConnection();
         PreparedStatement pstmt = conn.prepareStatement(query)) {
        
        pstmt.setString(1, username);
        pstmt.setString(2, password); // Note: In reality, passwords should be hashed!
        
        try (ResultSet rs = pstmt.executeQuery()) {
            return rs.next();
        }
    }
}
```
