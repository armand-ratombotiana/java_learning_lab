# Deep Dive: Advanced Polymorphism

## 1. Beyond Basic Overriding
Basic polymorphism teaches that a subclass can override a superclass method to provide specific behavior. However, Java's polymorphic engine is much more sophisticated, handling covariant return types, method hiding, and complex resolution rules when generics and interfaces are involved.

## 2. Covariant Return Types
Since Java 5, overriding methods are allowed to change the return type, *provided* the new return type is a subtype of the original return type. This is called a covariant return type.

```java
class Animal {
    public Animal reproduce() { return new Animal(); }
}

class Dog extends Animal {
    @Override
    public Dog reproduce() { return new Dog(); } // Valid! Dog is a subtype of Animal
}
```
*   **Why it matters**: It prevents the caller from having to cast the result when they are working with the specific subclass type.

## 3. Method Hiding (Static Polymorphism)
Polymorphism only applies to *instance* methods. If a subclass defines a static method with the same signature as a static method in the superclass, the method in the subclass *hides* the one in the superclass.

```java
class Parent {
    static void print() { System.out.println("Parent"); }
}
class Child extends Parent {
    static void print() { System.out.println("Child"); }
}

// Usage:
Parent p = new Child();
p.print(); // Prints "Parent"! The reference type determines which static method is called.
```
*   **Key Concept**: Instance methods are resolved at runtime based on the *actual object type* (Dynamic Binding). Static methods (and variables) are resolved at compile time based on the *reference type* (Static Binding).

## 4. Method Resolution Order (MRO) and Default Methods
With the introduction of `default` methods in interfaces (Java 8), a class can inherit multiple default methods with the same signature from different interfaces. Java uses strict rules to resolve these conflicts:

1.  **Classes Win**: A method defined in a class (or superclass) always wins over a default method in an interface.
2.  **Most Specific Interface Wins**: If `Interface B` extends `Interface A`, and both provide a default method, the method in `B` wins because it is more specific.
3.  **Manual Resolution**: If neither interface extends the other, the compiler throws an error. You MUST override the method in the implementing class and manually choose which interface's method to call using `InterfaceName.super.methodName()`.

```java
interface A { default void doWork() { System.out.println("A"); } }
interface B { default void doWork() { System.out.println("B"); } }

class C implements A, B {
    @Override
    public void doWork() {
        A.super.doWork(); // Explicitly choosing A's implementation
    }
}
```

## 5. Generic Polymorphism and Type Erasure
Generics introduce a complex layer to polymorphism. Because Java uses Type Erasure, `List<String>` and `List<Integer>` become just `List` at runtime.

### Bridge Methods
When a class implements a generic interface and specifies the type, the compiler must generate a synthetic "bridge method" to preserve polymorphism at runtime.

```java
interface Node<T> { void setData(T data); }

class StringNode implements Node<String> {
    public void setData(String data) { ... }
}
```
After compilation, `StringNode` actually has *two* `setData` methods:
1.  `public void setData(String data)` (Your code)
2.  `public void setData(Object data)` (The Bridge Method generated by the compiler, which casts `data` to `String` and calls the first method).
