# Interview Preparation: Advanced Polymorphism

This document covers advanced questions related to overriding, overloading, static binding, and generic type erasure.

## Q1: Can you override a `static` method in Java? What happens if you try?
**Answer:**
No, you cannot override a `static` method. Polymorphism (overriding) relies on dynamic binding at runtime based on the actual object type. Static methods are bound at compile time based on the reference type.
If you declare a static method in a subclass with the same signature as a static method in the superclass, it is called **Method Hiding**. If you call the method using a superclass reference, the superclass's method executes, even if the actual object is an instance of the subclass.

## Q2: What is a Covariant Return Type, and why is it useful?
**Answer:**
A covariant return type allows a method overriding a superclass method to return a narrower (more specific) type than the method it overrides. For example, if `Parent.get()` returns `Number`, `Child.get()` can return `Integer`.
It is highly useful in patterns like the Builder pattern or `clone()`, as it prevents the caller from having to explicitly cast the returned object to the specific subclass type when working with subclass references.

## Q3: What is the "Diamond Problem", and how does Java 8+ solve it with Interfaces?
**Answer:**
The Diamond Problem occurs in multiple inheritance when a class inherits from two superclasses that both define a method with the same signature. The compiler doesn't know which implementation to use.
Java avoids this with classes (single inheritance only). However, Java 8 introduced `default` methods in interfaces, re-introducing the possibility of the Diamond Problem if a class implements two interfaces with the same default method.
Java solves this by forcing the developer to resolve the ambiguity. The compiler will throw an error, and the implementing class *must* override the method. Inside the overridden method, the developer can choose which interface's implementation to call using `InterfaceName.super.methodName()`.

## Q4: Explain what a "Bridge Method" is and why the Java compiler generates it.
**Answer:**
Bridge methods are synthetic methods generated by the Java compiler to preserve polymorphism when using generics, due to Type Erasure.
If you have `interface Node<T> { void setData(T data); }` and `class StringNode implements Node<String> { void setData(String data) {} }`, type erasure changes the interface method to `setData(Object)`. 
However, `StringNode` only has `setData(String)`. To make sure that polymorphism works (e.g., when calling `setData` via a `Node` reference), the compiler generates a bridge method in `StringNode`: `void setData(Object data) { this.setData((String) data); }`.

## Q5: Why is it dangerous to call an overridable method from a constructor?
**Answer:**
When an object is instantiated, the superclass constructor runs *before* the subclass constructor.
If the superclass constructor calls a method that the subclass has overridden, the subclass's version of the method will execute. However, because the subclass constructor hasn't run yet, the subclass's instance variables have not been initialized. If the overridden method tries to access those variables, it will likely result in a `NullPointerException` or read default values (e.g., 0 or false), leading to unpredictable behavior.
