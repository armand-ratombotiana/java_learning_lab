# Deep Dive: Type Erasure

## 1. What is Type Erasure?
When Java 5 introduced Generics, the designers faced a massive constraint: **Backward Compatibility**. They needed generic code compiled in Java 5 to run seamlessly on older Java 1.4 Virtual Machines, which knew nothing about generics.

To achieve this, they implemented **Type Erasure**.
Generics in Java are strictly a *compile-time* construct. Once the compiler verifies that your code is type-safe, it completely **erases** all generic type information from the compiled bytecode (`.class` files). The JVM at runtime has almost no knowledge of the generic parameters you used in your source code.

## 2. How Erasure Works Under the Hood
The compiler performs three main actions during erasure:

1.  **Replace Type Parameters**: 
    *   If a type parameter is unbounded (`<T>`), it is replaced with `Object`.
    *   If a type parameter is bounded (`<T extends Number>`), it is replaced with the first bound (`Number`).
2.  **Insert Casts**: The compiler inserts type casts where necessary to preserve type safety.
3.  **Generate Bridge Methods**: The compiler generates synthetic methods to preserve polymorphism in extended generic classes.

### Example: Before Erasure (Source Code)
```java
public class Box<T> {
    private T item;
    public void set(T item) { this.item = item; }
    public T get() { return item; }
}

Box<String> stringBox = new Box<>();
stringBox.set("Hello");
String s = stringBox.get();
```

### Example: After Erasure (Bytecode equivalent)
```java
public class Box { // <T> is gone
    private Object item; // T replaced by Object
    public void set(Object item) { this.item = item; }
    public Object get() { return item; }
}

Box stringBox = new Box();
stringBox.set("Hello");
String s = (String) stringBox.get(); // Compiler inserted the cast!
```

## 3. The Consequences of Erasure
Because generic types do not exist at runtime, several intuitive operations are strictly forbidden by the compiler:

*   **Cannot instantiate generic types**: `new T()` is illegal because at runtime, `T` is just `Object`.
*   **Cannot create generic arrays**: `new T[10]` or `new List<String>[10]` is illegal.
*   **Cannot use `instanceof` with parameterized types**: `if (obj instanceof List<String>)` is illegal because at runtime, it's just a `List`. You can only check against the raw type: `if (obj instanceof List<?>)`.
*   **Cannot catch generic exceptions**: `catch (MyException<T> e)` is illegal.

## 4. Bridge Methods (Preserving Polymorphism)
When a class extends a generic class or implements a generic interface and specifies the type, erasure can break method overriding.

```java
class Node<T> {
    public void setData(T data) { ... }
}
class StringNode extends Node<String> {
    @Override public void setData(String data) { ... }
}
```
After erasure, `Node` has a method `setData(Object)`. `StringNode` has a method `setData(String)`. Because the signatures don't match, `StringNode` is no longer overriding the parent method!

To fix this, the compiler generates a hidden **Bridge Method** in `StringNode`:
```java
// Synthetic Bridge Method generated by compiler
public void setData(Object data) {
    this.setData((String) data); // Casts and calls your actual method
}
```

## 5. Reification and Type Tokens
A "reified" type is a type whose type information is fully available at runtime (e.g., primitives, non-generic classes like `String`, and raw types like `List`). Generic types like `List<String>` are *non-reifiable*.

If you need to know the generic type at runtime (e.g., for JSON deserialization or dependency injection), you must use a **Type Token**. You explicitly pass the `Class` object as a parameter.

```java
// Using a Type Token to bypass erasure limitations
public <T> T createInstance(Class<T> type) throws Exception {
    return type.getDeclaredConstructor().newInstance(); // Safe instantiation!
}

String s = createInstance(String.class);
```
