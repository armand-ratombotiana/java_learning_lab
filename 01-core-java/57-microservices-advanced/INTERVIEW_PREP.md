## 📝 Conceptual Questions

### Q1: What is the Dual-Write Problem, and how is it solved?
**Answer**:
The dual-write problem occurs when a microservice needs to update a database and publish a message to a message broker (e.g., Kafka/RabbitMQ) as part of the same business transaction. Because the database and the message broker are two distinct, independent systems, you cannot wrap them in a standard local ACID transaction. If the JVM crashes after writing to the database but before sending the message, the system is left in an inconsistent state.
It is solved using the **Transactional Outbox Pattern** (writing the event to an outbox table in the same database transaction, then polling that table asynchronously) or by using **Event Sourcing** (where the event *is* the database truth).

### Q2: Compare Saga Choreography vs Saga Orchestration.
**Answer**:
- **Choreography**: Is completely decentralized. Services emit events, and other services react to them independently. It is highly decoupled and requires no central controller. However, for complex workflows involving 4+ services, it becomes extremely difficult to understand the overall business flow (the logic is scattered), and implementing compensating transactions for failures is complex to track.
- **Orchestration**: Uses a central "Saga Orchestrator" service. This service explicitly issues commands to downstream services and waits for replies. If a step fails, the orchestrator issues the compensating commands. It centralizes the business workflow, making it easy to monitor and debug, but introduces a single point of failure/bottleneck and tighter coupling.

### Q3: How does the Bulkhead pattern differ from the Circuit Breaker pattern?
**Answer**:
- **Circuit Breaker**: Detects when a downstream service is failing and *stops sending requests* to it, returning an immediate error (failing fast) to give the downstream service time to recover.
- **Bulkhead**: Actively partitions system resources (like connection pools or thread pools) *before* a failure occurs. If Service A talks to DB1 and DB2, and uses a single global thread pool, a slowdown in DB1 will consume all threads, breaking DB2 access. A bulkhead gives DB1 and DB2 their own isolated, smaller thread pools, ensuring a failure in one does not sink the entire application.

## 💻 Whiteboarding Scenarios

### Scenario 1: The Anti-Corruption Layer (ACL)
**Problem**: Your company acquired a competitor. Your clean, Domain-Driven microservice (which uses `UUID`s and modern JSON records) needs to integrate with their 20-year-old SOAP-based CRM, which uses integer IDs and a messy, 50-field XML payload. How do you integrate them without ruining your clean domain model?

**Solution**:
Draw a boundary between the two systems. Do not let the XML classes generated by tools like JAXB cross into your business logic layer.
Implement an **Anti-Corruption Layer (ACL)**. This layer acts as an Adapter.
1. The ACL defines an interface in terms of *your* modern domain.
2. The implementation of that interface makes the SOAP call, receives the messy XML, and parses it.
3. The ACL then maps/translates the XML data into your clean `User` domain entity.
4. Your business logic only ever sees the clean `User` entity. If the legacy CRM is finally retired next year, you only have to rewrite the ACL implementation; your core business logic remains untouched.
