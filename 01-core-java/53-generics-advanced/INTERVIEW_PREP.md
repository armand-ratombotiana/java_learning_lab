# Interview Preparation: Advanced Generics

This document covers advanced questions related to PECS, Type Erasure, Bridge Methods, and the Self-Type Idiom.

## Q1: Explain the PECS rule in Java Generics. Give an example of when you would use `? super T`.
**Answer:**
PECS stands for **Producer Extends, Consumer Super**. It dictates how to use wildcards to maximize API flexibility.
*   If a collection is a **Producer** (your method only reads data from it), use `? extends T`. This guarantees you can read `T`, but prevents you from adding anything to it (since the compiler doesn't know the exact subtype).
*   If a collection is a **Consumer** (your method only writes data to it), use `? super T`. This guarantees you can safely add `T` to it, but you can only read `Object` from it.
**Example**: If you write a method that adds a default `User` to a list, the signature should be `void addDefaultUser(List<? super User> list)`. This allows the caller to pass a `List<User>` or a `List<Object>`, increasing the method's utility.

## Q2: What is the "Self-Type Idiom" (CRTP), and what problem does it solve in Java?
**Answer:**
The Self-Type Idiom (Curiously Recurring Template Pattern) looks like this: `abstract class Builder<SELF extends Builder<SELF>>`.
It solves the problem of method chaining (the Builder pattern) across inheritance hierarchies. If a subclass inherits a method from a base class, that method normally returns the base class type. If you chain it, you lose the subclass type context and cannot call subclass-specific methods.
By using the Self-Type idiom, the base class methods can return the generic type `SELF`. The concrete subclass implements an abstract method to return `this`, ensuring that inherited methods return the exact subclass type, preserving the fluent chain.

## Q3: Explain what a "Bridge Method" is and why the compiler generates it.
**Answer:**
Bridge methods are synthetic methods generated by the Java compiler to preserve polymorphism in the presence of Type Erasure.
If a class implements a generic interface (e.g., `class StringNode implements Node<String>`), type erasure changes the interface method signature to use `Object` (e.g., `setData(Object)`). However, the implementing class only has `setData(String)`.
To ensure that dynamic binding works when calling `setData` via a `Node` reference, the compiler generates a hidden "bridge method" in `StringNode`: `void setData(Object data) { this.setData((String) data); }`.

## Q4: Why does `List<String>` not extend `List<Object>`, even though `String` extends `Object`?
**Answer:**
Because Java generics are **invariant**.
If `List<String>` were a subtype of `List<Object>`, you could do this:
```java
List<String> strings = new ArrayList<>();
List<Object> objects = strings; // If this were allowed...
objects.add(new Integer(5));    // You just put an Integer into a List of Strings!
String s = strings.get(0);      // ClassCastException at runtime
```
To prevent this catastrophic type safety failure, generics are invariant. To achieve polymorphic behavior with generic collections, you must use wildcards (`? extends Object`).

## Q5: What is a Recursive Type Bound, and where is it most commonly used?
**Answer:**
A recursive type bound is when a type parameter is bounded by an expression that includes itself: `<T extends Comparable<T>>`.
It is most commonly used to ensure that a type can be compared to itself. It prevents a developer from creating a class `Apple` that implements `Comparable<Orange>`. The recursive bound forces `Apple` to implement `Comparable<Apple>`, ensuring type-safe sorting and comparison algorithms.
