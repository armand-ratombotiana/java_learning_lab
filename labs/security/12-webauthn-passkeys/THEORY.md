# Theory: 12-webauthn-passkeys

## Core Concepts

### WebAuthn Overview

WebAuthn (Web Authentication) is a W3C standard for passwordless authentication using public key cryptography. It enables secure authentication through biometrics, hardware tokens, and platform authenticators without sharing secrets over the network.

### FIDO2 Stack

The FIDO2 project consists of:
1. **WebAuthn (W3C)**: JavaScript API for web applications
2. **CTAP2 (FIDO)**: Client to Authenticator Protocol for external authenticators
3. **CTAP1 (FIDO U2F)**: Legacy protocol for U2F tokens

### Key Concepts

1. **Credentials**: Public key pairs generated by authenticators
   - **Non-Resident Key**: Key stored on server, credential ID on authenticator
   - **Resident Key (Discoverable Credential)**: Full key stored on authenticator

2. **Authenticators**: Devices that generate and use key pairs
   - **Platform Authenticator**: Built into device (TPM, Secure Enclave)
   - **Cross-Platform Authenticator**: External device (USB key, phone)

3. **Attestation**: Proving authenticity of the authenticator
   - None: No attestation
   - Basic: Using an attestation certificate
   - Self: Using the generated key itself
   - Privacy CA: Using a privacy-preserving certificate authority
   - ECDAA: Direct anonymous attestation using elliptic curves

4. **User Verification Methods**:
   - userPresence (UP): User touched the authenticator
   - userVerification (UV): User verified via PIN or biometric
   - Both can be combined for multi-factor

### Registration Flow

1. RP generates challenge and sends to client
2. Client passes to authenticator
3. Authenticator generates key pair
4. Public key and attestation sent to RP
5. RP stores public key and credential ID

### Authentication Flow

1. RP generates challenge and sends to client
2. Client discovers credential or RP specifies credential ID
3. Authenticator signs challenge with private key
4. RP verifies signature with stored public key

### Passkeys

Passkeys are discoverable credentials that sync across devices:
- Stored in platform keychain (iCloud Keychain, Google Password Manager)
- Synchronized via end-to-end encryption
- Available across all user devices
- Replace passwords entirely
- Phishing-resistant by design (bound to origin)

### Security Properties

1. **Phishing Resistance**: Credentials bound to RP origin
2. **No Shared Secrets**: Private key never leaves authenticator
3. **Replay Protection**: Challenge-response with origin binding
4. **Multi-Factor**: Something you have (authenticator) + something you are/know (biometric/PIN)
5. **Credential Privacy**: Attestation can be made privacy-preserving

### Implementation Considerations

- Challenge generation with sufficient entropy (32+ bytes)
- Proper RP ID configuration (origin validation)
- Credential ID management and storage
- User verification preferences (discouraged, preferred, required)
- Attestation verification policies
- Backup eligibility for passkey sync
- Extension support (appid, credProps, hmac-secret)
- Error handling for authenticator failures
