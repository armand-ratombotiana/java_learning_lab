# Theory: 19-siem-monitoring

## Core Concepts

### SIEM Fundamentals

Security Information and Event Management (SIEM) systems provide real-time analysis of security alerts generated by network hardware and applications. SIEM centralizes log data from multiple sources for threat detection, investigation, and compliance reporting.

### SIEM Architecture

1. **Collection Layer**: Log shippers (Filebeat, Winlogbeat, Syslog-ng)
2. **Processing Layer**: Log parsing and normalization (Logstash, Fluentd)
3. **Storage Layer**: Data indexing and search (Elasticsearch, OpenSearch)
4. **Correlation Layer**: Rule engine for threat detection
5. **Presentation Layer**: Dashboards and alerts (Kibana, Grafana)
6. **Response Layer**: Automated incident response

### Log Collection and Normalization

1. **Log Sources**: Web servers, databases, firewalls, cloud APIs, endpoints
2. **Log Formats**: JSON, syslog, Windows Event Log, CEF, LEEF
3. **Normalization**: Converting diverse formats to common schema
4. **ECS (Elastic Common Schema)**: Standardized field mapping
5. **Enrichment**: Adding context (geo IP, threat intel, user identity)

### Correlation Rules

Types of correlation rules:
1. **Threshold Rules**: Alert when event count exceeds threshold
2. **Temporal Rules**: Pattern of events over time window
3. **Sequence Rules**: Events occurring in specific order
4. **Trend Rules**: Deviation from baseline behavior
5. **Aggregation Rules**: Combining multiple event attributes

### Threat Detection Techniques

1. **Signature-Based**: Matching known IOC patterns
2. **Anomaly-Based**: Detecting deviation from baseline
3. **Behavior-Based**: UEBA for user and entity profiling
4. **Threat Intelligence**: Integrating external threat feeds
5. **ML-Based Detection**: Unsupervised learning for unknown threats

### Alerting Strategies

1. **Severity Classification**: Critical, High, Medium, Low, Info
2. **Escalation Paths**: Tier 1, Tier 2, Tier 3, Engineering
3. **Notification Channels**: Email, Slack, PagerDuty, SMS
4. **Alert Suppression**: Deduplication and grouping
5. **Triage Workflow**: Acknowledge, investigate, resolve, document

### Compliance Requirements

- **PCI DSS**: Logging requirements for cardholder data
- **SOC 2**: Audit logging and monitoring controls
- **HIPAA**: Access logging for protected health information
- **GDPR**: Data access and breach detection logging
- **SOX**: Financial system access monitoring

2. Privileged account abuse

5. Malware command and control traffic

7. Policy violation detection
8. Insider threat indicators

### Practical Application

Implementing this security mechanism involves:
1. Understanding the core protocol/mechanism thoroughly
2. Configuring the appropriate Spring Security modules
3. Testing with both valid and invalid inputs
4. Monitoring for security events and anomalies
5. Maintaining and updating as standards evolve

### Common Pitfalls

When implementing this security mechanism, avoid:
1. **Misconfiguration**: Incorrect settings can weaken security
2. **Missing Validation**: Not validating all inputs and outputs
3. **Improper Error Handling**: Revealing too much information
4. **Performance Ignorance**: Not considering security overhead
5. **Testing Gaps**: Incomplete test coverage for security paths

### Integration with Spring Security

Spring Security provides integration support through:
- Auto-configuration with sensible defaults
- Customizable filter chains for flexibility
- Authentication providers for various mechanisms
- Method security annotations for fine-grained control
- Testing utilities for security test cases

### Security Considerations

Always consider these security aspects:
1. **Defense in Depth**: Multiple independent security layers
2. **Least Privilege**: Minimum necessary permissions
3. **Secure Defaults**: Safe configuration out of the box
4. **Fail Secure**: Default to denying access on failure
5. **Audit Traceability**: Complete security event logging

### Performance Implications

Security mechanisms introduce performance overhead:
- Authentication: 5-50ms per request (depending on mechanism)
- Authorization: 1-10ms per request (cached policies)
- Encryption: 1-5ms per operation (AES-256 hardware accelerated)
- Audit Logging: 5-20ms per event (async batching recommended)
