# 09 — AWS Security — Theory

## Overview
AWS Security encompasses the tools and practices that protect cloud infrastructure, data, and applications. This lab covers IAM policies, KMS encryption, Security Groups, WAF, Shield, and GuardDuty.

## 1. IAM Policies and Roles

### Policy Structure
IAM policies are JSON documents that define permissions. Every policy contains:
- **Version**: Policy language version (2012-10-17 is current)
- **Statement**: Array of individual permission statements
- **Effect**: Allow or Deny
- **Action**: List of service operations (e.g., s3:GetObject)
- **Resource**: ARN(s) the policy applies to
- **Condition**: Optional constraints (IP range, time, MFA)

### Policy Types
- **AWS Managed Policies**: Pre-built policies maintained by AWS (e.g., AdministratorAccess)
- **Customer Managed Policies**: User-created policies for specific use cases
- **Inline Policies**: Policies embedded directly in a user, group, or role

### IAM Roles vs Users
- **Users**: Long-term credentials for people or services
- **Roles**: Temporary credentials obtained via STS AssumeRole
- **Service Roles**: Roles assumed by AWS services (e.g., EC2, Lambda)
- **Cross-Account Roles**: Roles that grant access across AWS accounts

### Least Privilege Principle
Grant only the permissions required to perform a task. Start with a minimal policy and add permissions as needed. Use IAM Access Analyzer to identify unused permissions.

### Policy Evaluation Logic
1. All policies attached to the principal are evaluated
2. Explicit DENY overrides any ALLOW
3. If no explicit ALLOW matches, the request is implicitly DENIED
4. Organizations SCPs act as a boundary on all accounts

## 2. KMS (Key Management Service)

### Key Types
- **Symmetric Keys**: Single key for encrypt and decrypt (most common)
- **Asymmetric Keys**: RSA or ECC key pairs for sign/verify and encrypt/decrypt
- **HMAC Keys**: For generating and verifying HMAC tags

### Key Hierarchy
```
AWS KMS Key (CMK)
  ├── AWS Managed (AWS services manage on your behalf)
  ├── Customer Managed (you control rotation, policy, deletion)
  └── Custom Key Store (CloudHSM-backed)
        └── Data Keys (generated by CMK for client-side encryption)
```

### Key Rotation
- Automatic rotation (annual for customer managed CMKs)
- On-demand rotation (creates new backing key, keeps old one for decryption)
- Imported key material cannot be automatically rotated

### Envelope Encryption
Encrypt data with a data key, then encrypt the data key with a CMK. This reduces KMS API calls and improves performance for large data.

### KMS Grants
Grants allow AWS services to use a CMK on your behalf without modifying key policies. They are time-limited and revocable.

## 3. Security Groups

### Stateful Firewall
Security Groups act as a virtual firewall at the instance level. They are stateful: if you allow inbound traffic, the response is automatically allowed regardless of outbound rules.

### Default Rules
- Inbound: Deny all traffic
- Outbound: Allow all traffic

### Rule Components
- Type: SSH, HTTP, HTTPS, Custom TCP/UDP, etc.
- Protocol: TCP, UDP, ICMP
- Port Range: Single port or range (e.g., 80, 443, 3000-3999)
- Source/Destination: CIDR block, security group ID, or prefix list

### Security Group vs NACL
| Feature | Security Group | NACL |
|---------|---------------|------|
| Level | Instance | Subnet |
| State | Stateful | Stateless |
| Rules | Allow only | Allow + Deny |
| Order | All rules evaluated | Priority order (lowest first) |
| Return traffic | Auto-allowed | Must be explicitly allowed |

## 4. WAF (Web Application Firewall)

### Core Features
- **Web ACLs**: Rules that filter web traffic based on conditions
- **Rate Limiting**: Block requests exceeding a threshold from a source IP
- **IP Sets**: Collections of IP addresses to allow or block
- **Regex Patterns**: Match patterns in request content
- **Managed Rule Groups**: Pre-configured rules from AWS and partners

### Common Rules
- SQL injection prevention
- Cross-site scripting (XSS) protection
- Geographic restriction
- Bot control
- Common vulnerabilities (OWASP Top 10)

### WAF Logging
Enable logging to S3, CloudWatch Logs, or Firehose for analysis. Logs contain full request headers, matched rules, and action taken.

## 5. Shield

### Shield Standard (Free)
- Automatic network-layer DDoS protection
- Protects against SYN/UDP floods, reflection attacks
- Available for all AWS resources

### Shield Advanced (Paid)
- Expanded DDoS protection at application layer
- 24/7 access to DDoS Response Team (DRT)
- Cost protection against scaling spikes
- Advanced metrics and visibility

### DDoS Attack Types
- **Volumetric**: Overwhelm bandwidth (UDP floods, amplification)
- **State Exhaustion**: Fill firewall/load balancer state tables
- **Application Layer**: Slow loris, HTTP floods targeting specific endpoints

## 6. GuardDuty

### Threat Detection Types
- **Reconnaissance**: Port scanning, brute force attempts
- **Instance Compromise**: Crypto mining, outbound malware traffic
- **Account Compromise**: Unusual API calls, credential exfiltration
- **S3 Threat Detection**: Data exfiltration, unusual access patterns

### Data Sources Analyzed
- VPC Flow Logs
- DNS Logs
- CloudTrail Management and Data Events
- CloudWatch Logs (S3 and EKS audit logs)

### Findings Severity
- **High**: Active compromise detected (e.g., crypto mining)
- **Medium**: Suspicious behavior (e.g., port probing)
- **Low**: Informational (e.g., service discovery)

### Automation
Use EventBridge rules to respond to GuardDuty findings: trigger Lambda for automatic remediation, send to SIEM, or create Jira tickets.

## Key Takeaways
1. IAM is the foundation of AWS security — master policies and roles
2. KMS provides centralized key management with envelope encryption
3. Security Groups are stateful instance firewalls
4. WAF protects web applications at Layer 7
5. Shield defends against DDoS attacks at multiple layers
6. GuardDuty continuously monitors for threats using ML and anomaly detection
