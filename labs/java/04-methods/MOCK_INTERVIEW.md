# Mock Interview Transcript: Methods

## Interviewer: Senior SWE, Oracle
## Candidate: Mid-level Java developer
## Time: 35 minutes
## Focus: Method design, overloading, varargs, parameter passing

---

**Q1: Is Java pass-by-value or pass-by-reference?**

**Candidate**: Java is always pass-by-value. For primitives, you pass a copy of the value. For objects, you pass a copy of the reference — so you can modify the object's state but not change which object the caller's variable refers to.

**Interviewer**: Prove it. Write a method that tries to swap two objects.

**Candidate**: 
```java
void swap(Object a, Object b) {
    Object temp = a;
    a = b;
    b = temp;
}
// After call: original references are unchanged because the copies were swapped, not the originals.
```

**Interviewer**: Right. What about this:

```java
void append(List<String> list) {
    list.add("world");  // This modifies the caller's list
    list = new ArrayList<>();  // This doesn't affect the caller
}
```

**Candidate**: Exactly. The first line modifies the object pointed to by the copy of the reference. The second line reassigns the local copy to a new object — the caller's reference still points to the original.

**Interviewer**: Let's talk about method overloading. Will this compile?

```java
void foo(int x) { }
void foo(long x) { }
```

**Candidate**: Yes. These have different parameter types.

**Interviewer**: What about this?

```java
void foo(int... x) { }
void foo(int[] x) { }
```

**Candidate**: No, that won't compile. Varargs (`int...`) is syntactic sugar for an array. The JVM sees both as having the same descriptor `([I)V`, so they're considered the same method.

**Interviewer**: How does overload resolution work when there are multiple candidates?

```java
void foo(int x) { }
void foo(long x) { }
void foo(Integer x) { }
foo(5);  // Which is called?
```

**Candidate**: Java picks the most specific one. `int` is chosen over `long` (no widening needed) and over `Integer` (autoboxing has lower priority than exact match). The order is: exact match → widening → autoboxing → varargs.

**Interviewer**: Good. What about method overriding — what's the rule for covariant return types?

**Candidate**: In Java 5+, an overriding method can return a subtype of the original return type. For example, if the parent method returns `Number`, the child can return `Integer`. This is called a covariant return type.

**Interviewer**: How does the JVM implement this? The bridge method concept?

**Candidate**: The compiler generates a synthetic bridge method with the parent's return type that calls the child's method. So the generated bytecode has:
```java
// Synthetic bridge
public Number foo() { return this.foo(); }  // calls Integer foo()
```
This maintains binary compatibility while allowing the covariant return.

**Interviewer**: Final question: when would you use a static factory method instead of a constructor?

**Candidate**: Several cases: (1) when the method has a meaningful name like `BigDecimal.valueOf()`, (2) when you want to return a cached object (like `Integer.valueOf()`), (3) when you want to return a subtype, (4) when construction could fail and you prefer returning null vs throwing, (5) for generic type inference with complex generics.
