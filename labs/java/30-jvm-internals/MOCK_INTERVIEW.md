# Mock Interview Transcript: JVM Internals

## Interviewer: Staff Engineer, Oracle
## Candidate: Senior Java developer
## Time: 50 minutes
## Focus: Runtime data areas, class loading, execution engine

---

**Q1: Describe the JVM runtime data areas. What is the difference between PC Register and native method stack?**

**Candidate**: JVM has per-thread and shared areas. Per-thread: (1) PC Register ŌĆö address of current instruction. (2) Java Stack ŌĆö frames for each method call (local variables, operand stack, frame data). (3) Native Method Stack ŌĆö for native (JNI) calls. Shared: (4) Heap ŌĆö objects, arrays, GC-managed. (5) Method Area (Metaspace in Java 8+) ŌĆö class metadata, bytecode, constant pool, static variables, JIT code.

**Interviewer**: Walk through what happens when you call `new MyClass()`.

**Candidate**: (1) Class loading: if MyClass isn't loaded, the class loader loads the .class file, verifies bytecode, prepares static fields, resolves references, and runs static initializers. (2) Memory allocation: JVM allocates memory on the heap for the new object (fast path uses TLAB ŌĆö Thread Local Allocation Buffer). (3) Zeroing: allocated memory is zeroed (not needed for TLAB). (4) Object header: set mark word (identity hash code, GC info, lock state) and klass pointer. (5) Constructor: call `<init>` which chains to parent constructors and initializes fields.

**Interviewer**: What is a TLAB and how does it improve allocation?

**Candidate**: TLAB (Thread Local Allocation Buffer) is a region of Eden space reserved for a single thread. The thread allocates objects in its TLAB without synchronization ŌĆö just a pointer bump. When the TLAB is exhausted, the thread requests a new one from the Eden space (synchronized). TLABs make allocation fast (~10 CPU instructions).

**Interviewer**: What is the code cache? What happens when it's full?

**Candidate**: The code cache stores compiled native code generated by the JIT compiler. If it fills up, the JIT stops compiling new methods. The JVM falls back to interpretation for uncompiled methods, causing severe performance degradation. Monitor with `-XX:+PrintCodeCache`. Increase with `-XX:ReservedCodeCacheSize=256m`. If you see "CodeCache is full", increase the size or reduce compilation.

**Interviewer**: Explain the Constant Pool in a class file.

**Candidate**: The constant pool is a table of constants used in the class: string literals, class names, method names, field descriptors, method handles, dynamic call sites. Each entry has a tag (CONSTANT_Utf8, CONSTANT_Class, CONSTANT_Methodref, etc.). The JVM resolves these symbolic references to direct references during linking.

**Interviewer**: How does the JVM determine if a class can be compiled by C1 or C2?

**Candidate**: Methods start in interpreter. The JVM counts method invocations and loop back-edges. The counters determine compilation: (1) -XX:CompileThreshold (default 1500 for C1, 10000 for C2) in non-tiered mode. (2) In tiered mode (default), methods progress: 0 (interpreter) ŌåÆ 2/3 (C1 with profiling) ŌåÆ 4 (C2). (3) OSR (On-Stack Replacement) compiles long-running loops mid-execution.

**Interviewer**: What is a safepoint? When do they occur?

**Candidate**: A safepoint is a point where all threads have stopped at a known, safe state (no modifications to the heap). Threads poll for safepoint requests during: (1) Method returns. (2) Back branches in loops. (3) Object allocation. The JVM uses safepoints for: GC pause start/stop, JIT code deoptimization, thread dump, biased locking revocation. `-XX:+PrintSafepointStatistics` for debugging.

**Interviewer**: Final: What is the difference between -Xms, -Xmx, -XX:MaxMetaspaceSize, and -Xss?

**Candidate**: `-Xms` ŌĆö initial heap size (start here, JVM may grow to Xmx). `-Xmx` ŌĆö maximum heap size. `-XX:MaxMetaspaceSize` ŌĆö maximum metaspace (infinite by default ŌĆö dangerous). `-Xss` ŌĆö thread stack size (default ~1MB per platform thread). Setting Xms = Xmx prevents JVM from spending time resizing the heap.
