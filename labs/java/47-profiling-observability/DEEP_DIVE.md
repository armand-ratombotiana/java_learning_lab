# Deep Dive: Profiling & Observability

## 1. async-profiler Deep Dive

async-profiler is the gold standard for Java profiling. It uses OS-level mechanisms for minimal overhead.

### Architecture

```java
// async-profiler works through three mechanisms:
//
// CPU profiling: perf_event_open (Linux) or kperf (macOS)
//   - Sets up hardware performance counters (PMC)
//   - Samples every N CPU cycles (default: every 1M cycles)
//   - No safepoint bias (unlike jstack-based profilers)
//   - Captures both Java and native stack traces
//
// Wall-clock profiling: Per-thread timers
//   - Uses SIGALRM or timer_create per thread
//   - Samples at fixed intervals (default: 10ms)
//   - Captures ALL threads, including sleeping/waiting
//   - Useful for latency analysis
//
// Allocation profiling: TLAB callback hook
//   - Hooks into TLAB allocation path
//   - Records stack traces at allocation sites
//   - No sampling: each allocation record is captured
//   - Can be filtered by size threshold

// async-profiler usage:
// ./profiler.sh start -e cpu <pid>
// ./profiler.sh stop -f profile.html <pid>
// ./profiler.sh start -e alloc -f alloc.jfr <pid>
// ./profiler.sh start -e wall -f wall.html <pid>
//
// Java API:
import java.io.*;
import java.lang.management.ManagementFactory;

public class AsyncProfilerAPI {
    public static void start() throws Exception {
        // Requires async-profiler on classpath or as agent
        // -agentpath:/path/to/libasyncProfiler.so=start,event=cpu,file=profile.html
        
        // Or dynamically attach:
        Class<?> profilerClass = Class.forName("one.profiler.AsyncProfiler");
        Object profiler = profilerClass.getMethod("getInstance").invoke(null);
        profilerClass.getMethod("start", String.class, String.class)
            .invoke(profiler, "cpu", "profile.html");
    }
}

public class ProfilingModes {
    // CPU profiling with event selection:
    // ./profiler.sh -e cpu <pid>           // CPU cycles
    // ./profiler.sh -e itimer <pid>        // ITIMER (virtual time)
    // ./profiler.sh -e cycles <pid>        // Hardware PMC cycles
    // ./profiler.sh -e instructions <pid>  // Hardware PMC instructions
    // ./profiler.sh -e cache-misses <pid>  // L2/L3 cache misses
    // ./profiler.sh -e branch-misses <pid> // Branch mispredictions
    
    // Wall clock profiling:
    // ./profiler.sh -e wall -t <pid>       // Sampled all threads
    // -t flag includes kernel threads
    // Wall clock shows where time is spent including I/O wait
    
    // Allocation profiling:
    // ./profiler.sh -e alloc <pid>          // All allocation sites
    // ./profiler.sh -e alloc=500k <pid>     // Only >500KB allocations
    // ./profiler.sh -e lock <pid>           // Lock contention
}
```

### Flame Graphs vs Flame Charts

```java
// Flame Graph (CPU):
//   X-axis: stack frequency (width = % of samples)
//   Y-axis: stack depth (call chain)
//   Color: random or by package
//   Use: identify hot methods (CPU bottlenecks)
//   Interactive: click to zoom, search for method
//   Generated by: profiler.sh -f profile.html
//
// Example flame graph interpretation:
//   ┌─────────────────────────────────────────────┐
//   │  java.util.HashMap::getNode (65%)           │  ← hottest
//   │  java.util.HashMap::get                      │
//   │  com.example.Cache::lookup                   │
//   │  com.example.Service::processRequest         │
//   └─────────────────────────────────────────────┘
//   Conclusion: HashMap.get is the bottleneck
//   Fix: use ConcurrentHashMap or increase initial capacity
//
// Flame Chart (Wall-clock):
//   X-axis: time (wall clock, not %)
//   Y-axis: stack depth
//   Color: by thread or package
//   Use: identify latency patterns (I/O waits, lock contention)
//   Interactive: select time range, identify patterns
//   
// Difference:
//   Flame Graph: aggregated over entire profile period
//   Flame Chart: time-series view, shows ordering of events
//   Profiling: use Flame Graph for CPU, Flame Chart for latency
//
// Iceberg Graph (allocation):
//   Like flame graph but shows allocation sites
//   Frame width = bytes allocated at that site
}
```

### async-profiler Events Deep Dive

```java
// Available events:
// CPU Events:
//   cpu           - CPU cycles (auto-selects PMC or itimer)
//   cycles        - Hardware cycles (PMC)
//   instructions  - Instructions retired
//   L1-dcache-load-misses
//   L1-dcache-loads
//   LLC-load-misses

//   branch-misses

//   alloc         - New object allocation
//   lock          - Java monitor contention
//   
// OS Events:
//   wall          - Wall-clock time
//   itimer        - ITIMER (virtual time, ~1ms resolution)
//
// Advanced: custom perf events
//   ./profiler.sh -e "perf::PERF_COUNT_HW_CPU_CYCLES" <pid>
//   ./profiler.sh -e "tracepoint:syscalls:sys_enter_write" <pid>

public class EventSelection {
    // Performance event selection guide:
    //
    // Problem | Event | Output
    // CPU-bound | cpu | Flame graph of hot methods
    // Memory-bound | LLC-load-misses | Cache miss hot spots
    // Allocation-heavy | alloc | Allocation sites
    // Slow I/O | wall | Latency hotspots
    // Lock contention | lock | Contended monitors
    // Branch mispredictions | branch-misses | Hot branches
}
```

## 2. JFR (Java Flight Recorder) Deep Dive

JFR is a built-in, low-overhead event recording system (originally from BEA JRockit, integrated into HotSpot since Java 11).

### JFR Architecture

```java
// JFR components:
// 1. Flight Recorder Engine (C++ in HotSpot)
//    - Thread-local event buffers (lock-free)
//    - Global buffer pool
//    - Data streaming thread
//    - Chunk-based file format
//    
// 2. JDK Flight Recorder API (jdk.jfr package)
//    - jdk.jfr.Event (base class for events)
//    - jdk.jfr.FlightRecorder (management)
//    - jdk.jfr.Recording (recording session)
//
// 3. JFR Event Types (50+ built-in events):
//    - jdk.GarbageCollection
//    - jdk.GCPhasePause
//    - jdk.ThreadSleep
//    - jdk.ThreadPark
//    - jdk.MethodSample
//    - jdk.StackTrace
//    - jdk.ObjectAllocationInNewTLAB
//    - jdk.ObjectAllocationOutsideTLAB
//    - jdk.ClassLoad
//    - jdk.ClassDefine
//    - jdk.LockInstance
//
// Overhead: <1% typically (target design goal)

import jdk.jfr.*;
import jdk.jfr.consumer.*;

public class JFRDeepDive {
    public static void listAllEvents() {
        // Programmatic query:
        for (EventType type : FlightRecorder.getFlightRecorder().getEventTypes()) {
            System.out.printf("%s (category: %s, enabled: %s)%n",
                type.getName(),
                String.join(", ", type.getCategoryNames()),
                type.isEnabled());
        }
    }
    
    // Example: creating a custom event
    @Label("Database Query")
    @Description("Database query execution")
    @Category({"Application", "Database"})
    @StackTrace(true)
    public static class DatabaseQueryEvent extends Event {
        @Label("Query")
        String query;
        
        @Label("Connection")
        String connectionId;
        
        @Label("Row Count")
        long rowCount;
        
        @Label("Duration (ms)")
        @Timespan(Timespan.MILLISECONDS)
        long duration;
    }
    
    // Using the custom event:
    public static void performQuery(String query, String connId) {
        DatabaseQueryEvent event = new DatabaseQueryEvent();
        event.begin();
        event.query = query;
        event.connectionId = connId;
        
        try {
            // ... execute query
            long rows = 42;
            event.rowCount = rows;
        } finally {
            event.commit();
        }
    }
}
```

### Custom Events Deep Dive

```java
// Custom JFR events — advanced features:

// 1. Threshold-based events (auto-write if duration exceeds threshold)
@Label("Slow Operation")
@Threshold("10 ms")
public class SlowOperationEvent extends Event {
    @Label("Operation Name")
    String operation;
}

// 2. Period events (fired at regular intervals)
@Label("Memory Snapshot")
@Period("1 s")
public class MemorySnapshotEvent extends Event {
    @Label("Heap Used")
    @DataAmount(DataAmount.BYTES)
    long heapUsed;
    
    @Label("Heap Max")
    @DataAmount(DataAmount.BYTES)
    long heapMax;
}

// 3. Event with custom values
@Label("Request Processing")
public class RequestEvent extends Event {
    @Label("URL")
    String url;
    
    @Label("Status")
    int status;
    
    @Label("Response Size")
    @DataAmount
    long responseSize;
    
    @Label("User")
    String userId;
}

// 4. Stack trace annotations
@Label("Critical Path")
@StackTrace(true)
@Enabled(true)
public class CriticalPathEvent extends Event {
    // Stack trace is automatically captured
}

// Thread-local event buffering:
// JFR writes events to a thread-local buffer (avoid synchronization)
// When buffer is full → flushed to global buffer pool
// When global buffer is full → written to disk
// The buffer size is critical for performance:
// -XX:FlightRecorderBufferSize=256k
// -XX:FlightRecorderMemorySize=10m
```

### JFR Streaming API

```java
// Since Java 14, JFR supports real-time streaming:
import jdk.jfr.consumer.*;

public class JFRStreaming {
    public static void main(String[] args) throws Exception {
        // Subscribe to specific events in real-time
        RecordingStream stream = new RecordingStream();
        
        // Enable garbage collection events
        stream.enable("jdk.GarbageCollection").withStackTrace();
        
        // Enable with threshold
        stream.enable("jdk.ThreadSleep").withThreshold("10 ms");
        
        // Custom events from our code
        stream.enable(DatabaseQueryEvent.class.getName());
        
        // Process events
        stream.onEvent("jdk.GarbageCollection", event -> {
            System.out.printf("GC pause: %.2f ms, cause: %s%n",
                event.getDuration().toMillis() / 1e6,
                event.getString("cause"));
        });
        
        stream.onEvent(DatabaseQueryEvent.class.getName(), event -> {
            System.out.printf("DB Query: %s (%d rows)%n",
                event.getString("query"),
                event.getLong("rowCount"));
        });
        
        // Event metadata access
        stream.onEvent(event -> {
            EventType type = event.getEventType();
            if (type.getName().contains("GC")) {
                // Process all GC events
            }
        });
        
        // Start streaming
        stream.startAsync();
        
        // Monitor for 60 seconds
        Thread.sleep(60000);
        stream.close();
    }
}
```

## 3. perf_event_open on Linux

Linux perf_event_open is the low-level interface used by async-profiler:

```java
// Perf event internals:
// - Opens a file descriptor for each performance counter
// - Events can be counted (sum) or sampled (profiling)
// - Each counter has a type and config:
//   type: PERF_TYPE_HARDWARE, PERF_TYPE_SOFTWARE, PERF_TYPE_TRACEPOINT
//   config: PERF_COUNT_HW_CPU_CYCLES, PERF_COUNT_HW_INSTRUCTIONS, etc.
//
// Sampling mode:
// - Set period (e.g., every 1000000 cycles)
// - Overflow signal (SIGPROF or SIGIO) delivered
// - Signal handler captures PC + stack trace
// - Requires /proc/sys/kernel/perf_event_paranoid ≤ 1
//
// async-profiler uses:
// - perf_event_open with sample_id_all
// - PERF_SAMPLE_IP | PERF_SAMPLE_TID | PERF_SAMPLE_CALLCHAIN
// - Call chain depth: -XX:MaxJavaStackTraceDepth=4096
//
// Container restrictions:
// - Docker: requires --cap-add SYS_ADMIN or --privileged
// - Or set /proc/sys/kernel/perf_event_paranoid = -1
// - Or use --all-user flag to avoid kernel address access

public class PerfEventSetup {
    public static void main(String[] args) {
        // Check perf_event_paranoid:
        // cat /proc/sys/kernel/perf_event_paranoid
        // 2 = allow only user-space events (default)
        // 1 = allow kernel events
        // 0 = allow discarding of kernel samples
        // -1 = no restrictions
        
        // async-profiler handles all this automatically
        // On macOS: uses kperf (Kernel Performance API)
        // On Windows: uses ETW (Event Tracing for Windows)
    }
}
```

## 4. Building a Profiling Dashboard

```java
// Combining JFR streaming + JMX for a real-time dashboard:
import jdk.jfr.consumer.*;
import javax.management.*;
import java.lang.management.*;
import java.util.concurrent.*;

public class ProfilingDashboard {
    private final ScheduledExecutorService scheduler;
    private final MBeanServer mbs;
    
    public ProfilingDashboard() {
        this.scheduler = Executors.newScheduledThreadPool(2);
        this.mbs = ManagementFactory.getPlatformMBeanServer();
    }
    
    public void start() throws Exception {
        // JFR stream
        RecordingStream stream = new RecordingStream();
        stream.enable("jdk.GarbageCollection");
        stream.enable("jdk.ThreadSleep").withThreshold("10 ms");
        stream.enable("jdk.JavaMonitorEnter").withThreshold("1 ms");
        
        stream.onEvent("jdk.GarbageCollection", event -> 
            System.out.printf("[ALERT] GC pause: %.1f ms%n", 
                event.getDuration().toNanos() / 1e6));
        
        stream.startAsync();
        
        // JMX polls
        scheduler.scheduleAtFixedRate(this::pollMetrics, 0, 5, TimeUnit.SECONDS);
        
        // Keep running
        Thread.sleep(3600000);
        stream.close();
    }
    
    private void pollMetrics() {
        try {
            // GC metrics
            for (GarbageCollectorMXBean gc : 
                 ManagementFactory.getGarbageCollectorMXBeans()) {
                long count = gc.getCollectionCount();
                long time = gc.getCollectionTime();
                System.out.printf("GC %s: %d collections, %dms%n",
                    gc.getName(), count, time);
            }
            
            // Memory pools
            for (MemoryPoolMXBean pool : 
                 ManagementFactory.getMemoryPoolMXBeans()) {
                MemoryUsage usage = pool.getUsage();
                double pct = 100.0 * usage.getUsed() / usage.getMax();
                System.out.printf("Memory %s: %.1f%% used%n",
                    pool.getName(), pct);
            }
            
            // Thread info
            ThreadMXBean threads = ManagementFactory.getThreadMXBean();
            System.out.printf("Threads: %d live, %d peak, %d started%n",
                threads.getThreadCount(),
                threads.getPeakThreadCount(),
                threads.getTotalStartedThreadCount());
                
        } catch (Exception e) {
            // Handle JMX errors silently
        }
    }
}
```

## 5. Profiling in Production

```java
// Production profiling best practices:
//
// 1. Always use async-profiler (not jstack/safepoint-based)
// 2. JFR recording with streaming (don't dump huge files)
// 3. Start with low overhead settings, increase as needed
// 4. Profile during representative load, not idle
//
// Emergency profiling:
//   jcmd <pid> JFR.start name=profile settings=profile duration=60s
//   jcmd <pid> JFR.dump name=profile filename=emergency.jfr
//
// Continuous profiling:
//   async-profiler in "loop" mode:
//     profiler.sh -d 30 -i 10ms -o jfr -f /data/profile.jfr <pid>
//   Rotate files, upload to observability platform
//
// Container-safe profiling:
//   -XX:+UseContainerSupport
//   -XX:ActiveProcessorCount=4
//   Use async-profiler with --all-user (no kernel addresses needed)
```

## 6. Benchmarking Profiling Overhead

```java
public class ProfilingOverheadBenchmark {
    private static final int WORK_UNITS = 10_000_000;
    
    // Compute-heavy workload
    public static double computeWorkload() {
        double sum = 0;
        for (int i = 0; i < WORK_UNITS; i++) {
            sum += Math.sin(i) * Math.cos(i) + Math.sqrt(i);
        }
        return sum;
    }
    
    public static void main(String[] args) throws Exception {
        // Warmup
        for (int i = 0; i < 5; i++) {
            computeWorkload();
        }
        
        // Run without profiler
        long t0 = System.nanoTime();
        double result1 = computeWorkload();
        long baseline = System.nanoTime() - t0;
        
        // Note: in practice, run with and without profiler attached
        // The profiler overhead is:
        //   (time_with_profiler - baseline) / baseline × 100
        
        System.out.printf("Baseline: %dms%n", baseline / 1_000_000);
        System.out.printf("Result: %f%n", result1);
    }
}
// Typical async-profiler overhead:
//   CPU profiling: 1-3%
//   Wall-clock: <1%
//   Allocation profiling: 5-15% (higher due to event processing)
//   JFR default: <1%
```

Profiling is ultimately about asking the right questions: where is my application spending time? Where is it allocating memory? Where is it waiting? The tools above answer these questions with minimal overhead.
