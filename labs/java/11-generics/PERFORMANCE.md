# Generics — Performance Characteristics

## Type Erasure: Zero Runtime Cost

Generics in Java have **zero runtime overhead** due to type erasure. All type checking happens at compile time; the generated bytecode is identical to what you'd write with explicit casts:

```java
// Generic:
List<String> list = new ArrayList<>();
String s = list.get(0);

// Equivalent erased bytecode:
List list = new ArrayList();
String s = (String) list.get(0);
```

## Cast Insertion

The compiler inserts casts at call sites. These casts are `checkcast` bytecode instructions — they take ~1-2 nanoseconds on modern JVMs. The JIT compiler can often elide redundant casts.

## Bridge Methods

Bridge methods add minimal overhead (one extra method call per invocation). The JIT typically inlines them.

## Impact on Collections

Generic collections like `ArrayList<T>` and `HashMap<K,V>` internally store `Object[]` or `Node[]`. The type parameter is erased — there is no type-specific code generation. This means no template bloat (unlike C++ templates) at the cost of boxing for primitives.

## Boxing Overhead

Primitives cannot be type parameters — `List<Integer>` boxes each `int` to `Integer`:

```java
List<Integer> list = new ArrayList<>();
list.add(42);  // Autoboxing: int → Integer
int x = list.get(0);  // Unboxing: Integer → int
```

Boxing overhead: ~3-5 nanoseconds per operation, plus garbage collection pressure. For performance-critical numeric code, consider primitive collections (e.g., `int[]`, Trove, Eclipse Collections).

## Specialization (or Lack Thereof)

Java generics do NOT specialize — unlike C++ templates or .NET generics. `ArrayList<Integer>` and `ArrayList<String>` use the exact same bytecode. The JIT, however, can see through erased types and optimize `ArrayList<Integer>` loops specifically.

## JIT Optimizations

The JIT compiler can:
- Devirtualize calls to final generic methods
- Inline bridge methods
- Eliminate bounds checks on generic array access (with escape analysis)
- Remove redundant `checkcast` instructions
- Specialize loops based on actual type frequency

## Performance Benchmarks

| Operation | Generic | Raw (with casts) | Difference |
|-----------|---------|------------------|------------|
| List.get() | ~5ns | ~5ns | None |
| Iteration (1000 items) | ~3μs | ~3μs | None |
| Map.put() | ~20ns | ~20ns | None |
| Bridge method dispatch | ~1ns | N/A | Negligible |

## When Performance May Differ

- **Array covariance**: `T[]` is not possible; use `Object[]` with casts
- **Instanceof**: `instanceof List<String>` not possible; use `List<?>`
- **Reflection**: `ParameterizedType` introspection is slower than direct access
- **Override bridges**: Each generic interface implementation generates a bridge method

## Best Practices

1. **Trust erasure**: No runtime performance penalty for generics
2. **Avoid boxing in hot paths**: Use `IntStream`, `LongStream`, primitive collections
3. **Prefer `@SuppressWarnings("unchecked")` over raw types**: Raw types suppress all checking but same performance
4. **Use `var` judiciously**: `var` preserves generic types; no performance impact
5. **Avoid reflection with generics**: `ParameterizedType` resolution is significantly slower than direct access
