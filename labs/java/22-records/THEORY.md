# Theory: Records in Java

## Theoretical Foundation

A record is a **nominal tuple** — a named type whose identity is defined by its state rather than by an identity reference. Formally, a record `R` with components `c₁, c₂, ..., cₙ` represents a **product type** in type theory: the set of all possible values of `R` is the Cartesian product of the sets of its component types.

```
R <: Product(C₁, C₂, ..., Cₙ)
value(R) = C₁ × C₂ × ... × Cₙ
```

This is distinct from a class, which typically represents an **abstract data type (ADT)** where the implementation is hidden behind an API. A record's representation is its API.

## The Record Contract

Every record automatically derives:

1. **A canonical constructor**: Takes all components as parameters, assigns them to `private final` fields
2. **Accessor methods**: Named the same as components (e.g., `record Point(int x, int y)` → `x()` and `y()`)
3. **`equals(Object)`**: Two records are equal if they are of the same type and all components are equal
4. **`hashCode()`**: Derived from all components (consistent with equals)
5. **`toString()`**: Returns a string containing the record name and all component values

## Immutability by Default

Records are **shallowly immutable**. The record itself cannot be changed after construction, but mutable component types (like `List`, `int[]`, `Date`) can still be modified through their own reference:

```java
record User(String name, List<String> tags) {}
var user = new User("Alice", new ArrayList<>());
user.tags().add("admin");  // Valid — the list reference is final but the list is mutable
```

## The Canonical Constructor

Every record has a canonical constructor generated by the compiler. This constructor:

1. Is `public` (unless a different access modifier is specified)
2. Takes the record components as parameters
3. Assigns component values to `private final` fields
4. Is implicitly `@SafeVarargs` if the last component is a varargs
5. Is `native` for the `clone()` method (records throw CloneNotSupportedException)

## Compact Constructors

A compact constructor omits the parameter list and allows validation or normalization before the implicit field assignment:

```java
record Range(int start, int end) {
    Range {
        if (start > end) throw new IllegalArgumentException("start > end");
        // Fields are assigned automatically after this block
    }
}
```

The compact constructor body is appended to the generated canonical constructor before the field assignments.

## Destructuring via Record Patterns

Records support **destructuring** through record patterns (Java 21):

```java
if (obj instanceof Point(int x, int y)) {
    // x and y are extracted automatically
}
```

This enables **nested destructuring**:

```java
if (obj instanceof Line(Point(int x1, int y1), Point(int x2, int y2))) {
    // Extract coordinates from nested records
}
```

## Local Records

Records can be declared **locally** (inside a method or block), which is useful for intermediate data transformations:

```java
public List<String> process(List<Item> items) {
    record ItemSummary(String name, int count) {}
    return items.stream()
        .collect(Collectors.groupingBy(Item::name, Collectors.summingInt(Item::quantity)))
        .entrySet().stream()
        .map(e -> new ItemSummary(e.getKey(), e.getValue()))
        .map(ItemSummary::toString)
        .toList();
}
```

## Serialization

Records have special serialization behavior:
- Serialization uses the canonical constructor (not `readObject()`/`writeObject()`)
- Deserialization always invokes the canonical constructor with the serialized components
- This guarantees that the invariant checks in the compact constructor are enforced during deserialization
- Records cannot be serialized through `readResolve()` or `writeReplace()` tricks

This makes records naturally secure against serialization attacks like the `HashSet` deserialization bug or `Builder` pattern attacks.
