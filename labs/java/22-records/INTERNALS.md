# Internal Implementation of Records

## Bytecode Representation

A record is compiled to a standard class file with specific attributes. The compiler adds:

1. **Record attribute**: A class file attribute that lists the record's components, their types, and their generic signatures
2. **`ACC_RECORD` flag**: A class access flag (0x0010) in the class file
3. **Enclosing method attribute**: For local records, to track the enclosing scope

The bytecode for a simple record `Point(int x, int y)` contains:

```
Class: Point
  flags: ACC_PUBLIC, ACC_FINAL, ACC_SUPER, ACC_RECORD
  super_class: java.lang.Record
  
  Field: x (I, ACC_PRIVATE, ACC_FINAL)
  Field: y (I, ACC_PRIVATE, ACC_FINAL)
  
  Method: void <init>(int, int)  // Canonical constructor
    ALOAD 0
    INVOKESPECIAL java/lang/Record.<init>()V
    ALOAD 0
    ILOAD 1
    PUTFIELD Point.x : I
    ALOAD 0
    ILOAD 2
    PUTFIELD Point.y : I
    RETURN
    
  Method: int x()
    ALOAD 0
    GETFIELD Point.x : I
    IRETURN
    
  Method: int y()
    ALOAD 0
    GETFIELD Point.y : I
    IRETURN
```

## The `java.lang.Record` Class

All records extend `java.lang.Record`, which provides:

```java
public abstract class java.lang.Record {
    protected Record() {}
    
    // No methods are declared here — they are all generated by the compiler
}
```

The class is abstract (you cannot directly instantiate `Record`), and the constructor is `protected` (called by generated constructors). The `Record` class itself has no methods — all methods (`equals`, `hashCode`, `toString`, accessors) are generated by the compiler.

## Reflection API

The reflection API supports records through:

```java
// Check if a class is a record
Class.isRecord()  // boolean

// Get record components
RecordComponent[] getRecordComponents()
    // Each RecordComponent has:
    //   getName() — component name
    //   getType() — component type (Class)
    //   getGenericType() — parameterized type
    //   getAccessor() — Method handle for the accessor method
    //   getAnnotations() — annotations on the component
```

## Serialization Internals

Record serialization uses special object stream classes:

1. When serializing a record, the `ObjectOutputStream` writes the record's class and the values of all components
2. When deserializing, `ObjectInputStream` reads the class and component values, then calls the canonical constructor via `MethodHandle` (NOT through reflection Field.set())
3. If the canonical constructor throws, deserialization fails — protecting against invalid states

The serialization implementation is in the JVM's core reflection layer, not in the record class itself. This is different from regular serialization where `readObject()` is invoked via reflection.

## Compiler-Specific Details

### Record Components vs. Fields
Record components are **not** fields from the annotation perspective. An annotation on a record component may be propagated to multiple targets (constructor parameter, field, accessor method) depending on the annotation's `@Target`. The compiler keeps track of these propagation rules.

### Generic Records
For generic records like `record Pair<A, B>(A first, B second) {}`, the compiler generates bridge methods for the accessors to handle type erasure:

```java
// Bridge for generic return type
public Object first() { return this.first; }  // bridge
public A first() { return this.first; }  // actual
```

## Memory Layout

Records currently use the same object header layout as regular classes (mark word, klass pointer, fields). Project Valhalla's value types may in the future allow records to be flattened (stored inline without an object header), reducing memory overhead.

Currently, a record like `record Point(int x, int y) {}` has:
- 12-16 byte object header (32-bit vs 64-bit JVM)
- 8 bytes for two int fields
- No alignment padding

Total: ~20-24 bytes per instance, plus heap overhead.

## Performance Optimizations

The JIT compiler can optimize record accessors aggressively:
- **Inlining**: Accessor methods are trivially inlinable
- **Scalar replacement**: Escape-analyzed records may be decomposed into individual fields on the stack
- **Dead field elimination**: Unused components may be eliminated

This means records can be as fast as direct field access in optimized code.
