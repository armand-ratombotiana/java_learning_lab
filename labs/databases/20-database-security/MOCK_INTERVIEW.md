# Mock Interview: Database Security (Lab 20)

**Role:** Database Security Engineer  
**Duration:** 40 minutes  
**Difficulty Progression:** Easy → Medium → Hard

---

## Round 1: Easy (5-10 minutes)

**Interviewer:** What are the main database security layers?

**Candidate:** Database security spans multiple layers:
1. **Network security:** Firewall, IP whitelist, VPN, TLS encryption for client-server
2. **Authentication:** Who can connect? Passwords, LDAP, Kerberos, SSL certificates
3. **Authorization:** What can each user do? Roles, privileges, grants
4. **Data at rest:** Encryption (TDE), encrypted backups
5. **Data in motion:** TLS/SSL, Oracle Net encryption
6. **Auditing:** Track who accessed what and when
7. **Row-level security:** VPD, Oracle Label Security
8. **Data masking:** Hide sensitive data in non-production

**Interviewer:** How does Oracle Transparent Data Encryption (TDE) work?

**Candidate:** TDE encrypts data at rest without application changes:
1. **Master encryption key** stored in Oracle Wallet (external security module)
2. **Table/table spaces encryption key** encrypted by master key, stored in data file header
3. When reading: Oracle decrypts on-the-fly using the key from wallet
4. When writing: Oracle encrypts before writing to disk
5. **Performance impact:** 5-10% overhead (CPU-based encryption), less with hardware acceleration

`CREATE TABLESPACE secure_data ENCRYPTION USING 'AES256' DEFAULT STORAGE(ENCRYPT);`

## Round 2: Medium (10-15 minutes)

**Interviewer:** Explain Oracle Virtual Private Database (VPD) and how to implement it.

**Candidate:** VPD enforces row-level security at the database level:
```sql
-- Create security policy function
CREATE OR REPLACE FUNCTION hr_sec(p_schema VARCHAR2, p_obj VARCHAR2) 
RETURN VARCHAR2 AS
BEGIN
    IF SYS_CONTEXT('USERENV', 'SESSION_USER') = 'HR_MANAGER' THEN
        RETURN 'org_id IN (SELECT org_id FROM manager_orgs WHERE manager = USER)';
    ELSIF SYS_CONTEXT('USERENV', 'SESSION_USER') = 'HR_EMPLOYEE' THEN
        RETURN 'employee_id = (SELECT employee_id FROM employees WHERE email = USER)';
    ELSE
        RETURN '1=0'; -- No access
    END IF;
END;

-- Apply policy to table
BEGIN
    DBMS_RLS.ADD_POLICY(
        object_name => 'employees',
        policy_name => 'hr_row_level',
        policy_function => 'hr_sec',
        statement_types => 'SELECT, INSERT, UPDATE, DELETE'
    );
END;
```

Every SQL query against the table automatically appends the WHERE clause generated by the function.

**Interviewer:** What is Oracle Database Vault?

**Candidate:** Database Vault restricts privileged user access:
- **Realm:** Protects sensitive objects from unauthorized access (even DBA)
- **Command rule:** Restricts SQL commands (e.g., prevent DROP TABLE during business hours)
- **Factor:** Multi-factor authentication for database access
- **Rule set:** Condition evaluation for access decisions

Use case: Prevent DBAs from seeing salary data while still allowing them to manage the database.

## Round 3: Hard (15-20 minutes)

**Interviewer:** Design a comprehensive database security framework for a healthcare application handling PHI (Protected Health Information) subject to HIPAA compliance.

**Candidate:** 

**Multi-layer security design:**

**Layer 1 — Network isolation:**
- Database in private VPC subnet, no direct internet access
- Application servers connect via private network endpoint
- TLS 1.3 for all connections
- Full network encryption (Oracle Net with AES-256)

**Layer 2 — Authentication:**
```sql
-- Multi-factor authentication (password + certificate)
CREATE USER patient_admin IDENTIFIED BY password;
ALTER USER patient_admin ENABLE SSL;

-- LDAP integration for centralized identity management
-- Oracle Enterprise User Security for single sign-on
```

**Layer 3 — Authorization:**
```sql
-- Role-based access control
CREATE ROLE patient_read_only;
CREATE ROLE patient_read_write;
CREATE ROLE patient_admin;

-- Row-level security based on clinic affiliation
CREATE FUNCTION phi_sec(...) RETURN VARCHAR2 AS
BEGIN
    -- Physician can see their own patients
    RETURN 'clinic_id IN (SELECT clinic_id FROM physician_assignments WHERE physician = USER)';
END;

-- Column-level encryption for SSN
CREATE TABLE patients (
    patient_id NUMBER,
    name VARCHAR2(100),
    ssn VARCHAR2(200) ENCRYPT 'AES256' -- TDE column encryption
);
```

**Layer 4 — Auditing (HIPAA requirement):**
```sql
-- Standard auditing
AUDIT SELECT, INSERT, UPDATE, DELETE ON patients BY ACCESS;

-- Fine-grained auditing (who accessed PHI)
BEGIN
    DBMS_FGA.ADD_POLICY(
        object_name => 'patients',
        policy_name => 'phi_access_audit',
        audit_column => 'ssn, diagnosis, treatment',
        audit_condition => 'USER != ''SYSTEM''',
        statement_types => 'SELECT'
    );
END;
```

**Layer 5 — Data protection:**
| Mechanism | Coverage |
|-----------|----------|
| TDE | All tablespaces |
| Backup encryption | RMAN encrypted backups |
| Network encryption | All connections |
| Data masking | Non-production copies |
| Database Vault | Protect PHI from privileged users |

**Layer 6 — Monitoring:**
- Real-time alerting on: multiple failed logins, data export patterns, unusual query patterns
- SIEM integration (Splunk) with database audit logs
- Weekly privilege review reports
