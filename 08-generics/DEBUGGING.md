# Debugging Generics Issues in Java

## Common Failure Scenarios

### Type Erasure Problems

Java generics use type erasure at runtime—all generic type information is removed. This creates fundamental limitations that cause unexpected behavior. A common issue is creating new instances of parameterized types, which is impossible at runtime. Code like `new T()` fails because the runtime has no knowledge of what T actually is. You must pass a Class<T> object or use reflection to construct instances.

Array type interactions with generics cause ClassCastException at unexpected points. A `List<String>` cannot be assigned to `List<Object>` because the compiler prevents the potential for runtime ClassCastException. However, you can assign `List<String>` to `List<?>` (unbounded wildcard), which is safe but restricts what you can do with the elements.

Bridge methods are generated by the compiler to maintain type compatibility between generic superclasses and non-generic subclasses. These synthetic methods can cause confusing stack traces that show methods you never wrote. The bridge method delegates from the signature-less superclass method to your implemented typed method.

### Bounded Type Issues

Wildcard bounds create subtle subtyping rules. `List<? extends Number>` means a list of some unknown type that extends Number—you can read Number from it but cannot add anything (except null). `List<? super Integer>` means a list of some unknown type that is a supertype of Integer—you can add Integer to it but can only read Object from it. This difference (producer extends, consumer super) is the PECS rule.

When combining multiple bounds, the order matters only syntactically, but the type constraints are AND semantics. `<T extends A & B & C>` means T must implement all three. With wildcards, you cannot have multiple upper bounds—`List<? extends A & B>` is a compile error. You must use a type that combines both in its hierarchy or use a wildcard with one bound and a method constraint.

## Debugging Techniques

### Diagnosing Type Erasure Issues

When facing erasure-related bugs, the key is recognizing that runtime type information is limited. Use `object.getClass()` to see the actual runtime type, but remember it won't include generic parameters. For debugging, add logging that prints the actual types involved—you'll often discover you're working with raw types when you expected parameterized ones.

The `Class<T>` token pattern is the standard solution for creating generic instances at runtime. Pass `Class<T>` into methods that need to construct T objects. For collections, the `ArrayList<T>` constructor accepts a Class parameter to help with type checking when adding elements.

To handle generic arrays, use `List<T>` instead of T[] when possible. If you must use arrays, create `Object[]` and cast, or use `Array.newInstance(Class<T>, size)` with reflection. Never use `new T[n]`—it won't compile.

### Resolving Wildcard Confusion

When wildcard types cause compiler errors, draw the type hierarchy and apply the PECS rule. Ask yourself: does the code produce values from the collection (read) or consume values into the collection (write)? Producer extends, consumer super applies to the method's perspective on the collection parameter.

Use the "capture conversion" pattern when you need to call methods that require specific types but only have wildcards. Write helper methods that "capture" the wildcard type by inferring it from the argument, then use those helpers to perform operations the wildcard would otherwise block.

## Best Practices

Use generics consistently throughout your codebase. Mixing raw types with parameterized types defeats the purpose of generics and introduces type safety holes. The compiler warns about raw types—don't ignore these warnings.

Prefer interfaces over concrete types in generic declarations. `List<T>` is more flexible than `ArrayList<T>` because it allows substitution with any List implementation.

Use `@SuppressWarnings("unchecked")` sparingly and only in methods where you've verified the type safety manually. Document why the code is safe—future maintainers need to understand why you suppressed the warning.

Prefer specific type bounds over unbounded wildcards when you can. `List<Number>` is more informative than `List<?>` and provides better type checking. Use `? extends Object` rather than raw `?` when you need to express "any type".
